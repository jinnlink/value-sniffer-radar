# Next: 从“提示”走向“可验证能赚钱”

这不是保证收益的系统；目标是把“机会”变成可验证的、可执行的、可淘汰的流水线。

## 1) 先把收益验证跑通（必须）

- 启用 `paper_log`（JSONL），把每条事件记录下来
- 做一个离线评估器（读取 JSONL + 拉取后续行情 + 统计收益/回撤/胜率/可执行率）
- 指标优先级：
  - 可执行率（有无足够流动性、价差是否吃掉优势）
  - 期望收益（扣成本后）
  - 最大回撤（你当前目标：5%）

## 2) “套利级”需要的前置（先模拟验证，再开户）

只有股票账户时，严格套利空间很小；更接近套利的方向通常至少需要其一：

- 期货账户：跨期价差 / 基差
- 期权账户：put-call parity / 合成多空 / box
- 两融（含融券）：可转债对冲、ETF成分对冲（坑多）

## 3) 数据分层（避免又慢又贵）

- 慢数据：Tushare（列表/日线/基本面）
- 快照（尽量实时）：券商行情/miniQMT（最稳）；free-first 可接公开快照源（要限流/降级）
- 统一“净优势”过滤：没有 bid/ask 只能做观察，不发“套利类”票

## 4) 自动下单的正确姿势（不走“点鼠标”偏方）

建议路线：
1) `event` 报警
2) `ticket` 下单草案（含风控参数、价格保护、失效条件）
3) 人工确认（先跑通）
4) 再接官方 API 做全自动执行 + 审计对账 + 一键熔断
